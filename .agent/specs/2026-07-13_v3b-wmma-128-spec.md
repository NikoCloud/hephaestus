# G1b-3b — 128×128 Multi-Wave WMMA GEMM (Implementation Spec)

**Status:** RATIFIED 2026-07-13 — 128×128 / 8-wave / BK=16. Intensity-64 step; **not** the final gate. Profiling is a hard gate this round.
**Owner of spec:** Opus. **Implementer:** Grok. **Target:** `src/hephaestus/wmma_gfx12.mojo` (v3b path, alongside v1/v2/v3a).
**Depends on:** v3a (`.agent/specs/2026-07-13_v3a-wmma-wide-tile-spec.md`) — same structure, wider tile. G1b-0 mappings.
**Gate:** G1b-3b — bitwise-correct 128×128 GEMM at intensity 64 (2× v3a), **plus measured VGPR/occupancy and GEMM/non-GEMM split** to choose the gate-closer.

---

## 0. Intent

v3a (64×64, intensity 32) hit 793 tok/s prefill (3.5× v2). v3b widens to **128×128** (intensity 64) — a clean one-variable step: same sub-row-per-wave ownership as v3a, just 8 sub-cols and 8 waves. Expected ~1.7–2× → **~1350–1500 tok/s**, honestly short of the G1b-3 gate (~2100). v3b's second job is to **stop guessing**: it must produce real VGPR/occupancy numbers and a GEMM-vs-non-GEMM time split, because the extrapolation says the gate is a multi-lever problem, and the next lever should be chosen on data.

Scope: one 128×128 tile, single-buffered LDS, BK=16, fused residual for o/down (unchanged from v3a). No double-buffering (v4), no 128×256 (v3c), no FP8 (step 8), no remainder masking.

---

## 1. Why 128×128 (not 128×256) — record for the successor

128×256 (intensity 85) is the *eventual* gate-relevant layer tile, but it is deferred to **v3c**, for three concrete reasons — none of which is "the dims don't divide":

- **Divisibility is fine for the layers, one exception at the tail.** Every transformer-layer N *is* ÷256 (q=4096, k/v=1024=256×4, o=2560=256×10, gate/up=9728=256×38, down=2560). The **only** ÷256 holdout is **lm_head N=151936** (÷128, not ÷256). lm_head is one tail matmul vs 36×7 layer matmuls, so it's a cheap dispatch fallback — not a blocker for v3c, just a carve-out.
- **Register pressure.** 128×256 = 128 sub-tiles; at 8 waves that's 16 acc/wave = 128 VGPR (occupancy cliff). It needs **16 waves** (512-thread workgroups) to get back to 8 acc/wave. That plus the lm_head carve-out plus per-matmul `BN` is three new variables at once.
- **128×128 is uniform.** All N (incl. lm_head, 151936÷128=1187) are ÷128 → one code path, no dispatch special-case, register-safe at ~87 VGPR without 512-thread workgroups.

So v3b takes the register-safe uniform step; v3c revisits 128×256-for-layers once v3b's occupancy data says whether 512-thread/16-wave workgroups are viable.

## 2. Configuration

| Param | v3a | **v3b** |
|---|---|---|
| BM × BN | 64 × 64 | **128 × 128** |
| Sub-tile grid (sr × sc) | 4 × 4 | **8 × 8 = 64** |
| Waves / workgroup | 4 | **8 (256 threads)** |
| Wave `w` owns | sr = w, all sc | **sr = w (rows w*16..w*16+15), all 8 sub-cols** |
| Acc / wave | 4 | **8** — `acc[sc]`, sc∈0..7, `SIMD[f32,8]` |
| VGPR/lane (est) | ~55 | **~87** (8×8 acc + frags + addr) |
| LDS / strip | 4 KB | **8 KB** (A 128×16 + B 128×16) |
| Intensity | 32 | **64** |
| LDS_STRIDE | 16 | 16 |
| BK | 16 | 16 |

---

## 3. Grid / block / tile coordinates

```
Grid  = ( ceildiv(N, 128), ceildiv(M, 128) )    # x = N-tiles, y = M-tiles
Block = 256 threads (8 waves × 32)
NB = blockIdx.x * 128
MB = blockIdx.y * 128
```

## 4. Thread indexing

```
tid = threadIdx.x           # 0..255  (cooperative load)
w   = tid / 32              # 0..7    wave = owned sub-row sr
l   = tid % 32              # 0..31   lane-in-wave (G1b-0 map, WMMA, store)
```

---

## 5. LDS layout + allocation

Both tiles `[128, 16]` row-major, stride `LDS_STRIDE`.
- `A_lds[row, col]` = A[MB+row, ks+col]  (row = M-index 0..127, col = K-index 0..15)
- `B_lds[row, col]` = W[NB+row, ks+col]  (row = **N**-index 0..127, col = K-index 0..15) — n-major, transpose baked in exactly as v2/v3a.

```mojo
comptime LDS_STRIDE = 16
var A_lds = stack_allocation[128 * LDS_STRIDE, BF16, address_space = AddressSpace.SHARED]()
var B_lds = stack_allocation[128 * LDS_STRIDE, BF16, address_space = AddressSpace.SHARED]()
var acc = InlineArray[SIMD[DType.float32, 8], 8](SIMD[DType.float32, 8](0))   # 8 accumulators, zeroed
```
LDS used: 2×(128×16) BF16 = 8 KB / workgroup (of 64 KB). Not the occupancy limiter — VGPR is (see §8).

---

## 6. The K-loop (per workgroup)

```
for ks in range(0, K, 16):        # BK = 16
    # ---- 6a. COOPERATIVE LOAD (all 256 threads) ----
    # A [128,16] = 2048 elems, 256 threads, 8 passes. Coalesced: 16 contiguous
    # 32-byte runs per pass (16 threads per row).
    for i in range(8):
        row = i*16 + tid//16                # 0..127
        col = tid % 16                      # 0..15
        A_lds[row*LDS_STRIDE + col] = A_global[(MB + row)*K + ks + col]

    # B tile (from W[N,K], n-major) [128,16], identical structure:
    for i in range(8):
        row = i*16 + tid//16                # N-index 0..127
        col = tid % 16                      # K-index 0..15
        B_lds[row*LDS_STRIDE + col] = W_global[(NB + row)*K + ks + col]

    barrier()                               # workgroup barrier: B shared across all 8 waves

    # ---- 6b. FRAGMENT LOAD + WMMA (reuse loop) ----
    # A fragment for THIS wave's sub-row (sr = w). Loaded ONCE, reused across 8 sc.
    var a = SIMD[DType.bfloat16, 8]()
    for j in range(8):
        a[j] = A_lds[(w*16 + l%16)*LDS_STRIDE + (l/16)*8 + j]

    for sc in range(8):                     # 8 N sub-cols
        var b = SIMD[DType.bfloat16, 8]()
        for j in range(8):
            b[j] = B_lds[(sc*16 + l%16)*LDS_STRIDE + (l/16)*8 + j]   # G1b-0 B-map, col base = sc*16
        acc[sc] = llvm_intrinsic["llvm.amdgcn.wmma.f32.16x16x16.bf16", SIMD[DType.float32, 8]](
            bitcast[DType.int16, 8](a),
            bitcast[DType.int16, 8](b),
            acc[sc],
        )

    barrier()                               # all reads done before next strip overwrites LDS
```

**Critical (same trap as v3a):** the `a` fragment load is hoisted OUT of the `sc` loop — that is the 8× A-reuse. Reloading `a` inside the `sc` loop compiles, passes correctness, and benchmarks like v2. Do not.

## 7. Store + fused residual epilogue (after the K-loop)

Unchanged from v3a except the sub-col count (8) and sub-row base. Residual added in **F32 before the BF16 cast**.

```mojo
for sc in range(8):
    for j in range(8):
        m = MB + w*16 + (l/16)*8 + j        # G1b-0 store row: sub-row base w*16
        n = NB + sc*16 + l%16               # G1b-0 store col: sub-col base sc*16
        var val = acc[sc][j]                # F32
        @parameter
        if FUSE_RESIDUAL:
            val += Float32(residual[m*N + n])   # residual stream x[m,n]
        C[m*N + n] = val.cast[out_dtype]()
```
- q/k/v/gate/up → `FUSE_RESIDUAL=False`.
- o_proj, down_proj → `FUSE_RESIDUAL=True`, `residual` = `acts.x`, N=2560.

---

## 8. Register & LDS budget

Per lane: 8 acc × 8 F32 = **64 VGPR**; `a` frag 4; `b` frag 4 (transient); addressing/loops ~15 → **~87 VGPR/lane**. This is the binding resource (LDS 8 KB is not). Whether ~87 VGPR gives acceptable occupancy on gfx1201 is **unknown and must be measured** (§10) — it's the input that decides whether v3c's 128×256/16-wave (which needs ~64 VGPR at higher LDS) is worth pursuing vs v4 double-buffering.

---

## 9. Correctness gate (identical structure to v3a §8)

- **Plain path** (q/k/v/gate/up, `FUSE_RESIDUAL=False`): v3b must be **bit-identical to `wmma_gemm_bf16_v1`**. Exact `==`.
- **Fused path** (o/down): bitwise reference is **`wmma_gemm_bf16_v1` + F32 residual-add-then-cast** (`cast_bf16(f32(v1) + f32(residual))`), **NOT** the naive `linear_add_residual`. Exact `==`.
- **vs naive `linear_add_residual`:** tolerance backstop only (`1e-5 + 1.6e-2·|ref|`); matmul order differs, so not bitwise.

Test shapes (M÷128, N÷128, K÷16): `128×128×64`, `128×256×64`, `512×4096×2560` (real q_proj prefill), `512×2560×9728` (real down_proj, fused), `512×151936×2560` (lm_head — confirms the uniform ÷128 path).

## 10. Profiling gate — HARD requirement this round (v3a skipped it)

Per the g1a2 principle, "real profiling, not guessing." Both must be captured and reported:

1. **rocprof: actual VGPR/lane and occupancy** (waves resident per SIMD/CU). Report the number; flag if occupancy is below ~2 waves/SIMD. This decides v3c viability.
2. **GEMM vs non-GEMM time split.** Measure `T_gemm` (the WMMA linear() calls) vs `T_other` (attention, RMSNorm, RoPE, residual, elementwise) across the prefill. This is the deciding datum: extrapolation from (v2=226, v3a=793) puts the non-GEMM floor near ~4800 tok/s and suggests even intensity-85 lands ~1650 — so the next lever (wider tile vs double-buffering vs non-GEMM optimization) must be chosen from this split, not guessed.

---

## 11. Verification checklist

1. **v3b plain vs v1 bitwise** — 0 mismatches, all §9 shapes (incl. lm_head 151936).
2. **v3b-fused vs (v1 + F32 residual) bitwise** — 0 mismatches, o/down shapes.
3. **Teacher-forced decode (4B)** — 255–256/256 (decode uses gemv; confirms no regression).
4. **Prefill 512 tok/s, 3 reps** — measure at M=512 (real forward pass). Honest expectation **~1350–1500** (intensity 64 + non-GEMM floor); report actual. This is an intensity step, **not** the G1b-3 gate.
5. **rocprof VGPR + occupancy** (§10.1).
6. **GEMM/non-GEMM split** (§10.2).

---

## 12. Preconditions & out of scope

**Preconditions:** M÷128, N÷128, K÷16. All real shapes qualify uniformly: M=512, N∈{4096,1024,2560,9728,151936} (all ÷128, incl. lm_head), K∈{2560,4096,9728}. Arbitrary prompt lengths (M not ÷128) → remainder masking, **deferred**.

**Do NOT build here:** double-buffering (v4), 128×256 / 16-wave / 512-thread (v3c), FP8 (step 8), remainder/edge tiles, BK>16, wave-assignment other than sub-row=w.

---

## 13. Gate definition (G1b-3b)

**PASS** = §11.1 ∧ §11.2 bitwise-exact ∧ §11.3 decode ok ∧ §11.4 prefill measured/reported ∧ §11.5 VGPR+occupancy captured ∧ §11.6 GEMM/non-GEMM split captured.
This is **not** G1b-3 (~2100). On pass: intensity-64 step banked; the §10 data selects the gate-closer — **v3c** (128×256-for-layers, 16-wave, lm_head fallback) if occupancy allows a wider tile, **v4** (double-buffering) if the barrier stalls dominate, or non-GEMM work if `T_other` is the floor. Record confirmed VGPR/occupancy in the kernel header for that decision.
