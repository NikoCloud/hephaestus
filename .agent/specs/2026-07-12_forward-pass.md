# Implementation Spec: BF16 Forward Pass (Phase 1a)
## Date: 2026-07-12
## Gate: G1a-1 (token-identical output vs HF reference, 3 prompts x 256 tokens)

## Context

The safetensors loader is done (G1a-3 PASS, bench/1a-load.md): both models load
into a single DeviceBuffer arena with TileTensor views over every weight.
This spec covers the next deliverable: the forward pass + greedy decode loop,
validated tiny-first against logit fixtures.

Read SPEC.md for scope law. Read docs/architecture-dossier.md for constants —
the four warnings in it are load-bearing here and are repeated below at the
point where each bites. Read docs/kernel-interface-report.md for kernel
contracts. All entry points below were re-verified to exist on 2026-07-12.

## Oracle fixtures (verified format, 2026-07-12)

Two fixture sets, same naming convention, same diff harness:

| | tiny (fixtures/tiny_random/) | 4B (fixtures/oracle/) |
|---|---|---|
| input IDs | reference_outputs.json `input_ids` | promptN_input_ids.json (flat list; 10/12/11 tokens) |
| expected tokens | reference_outputs.json `output_ids` (prompt + 16 new) | promptN_output_ids.json (**256 generated tokens only, prompt NOT included**) |
| step logits | oracle/promptN_logits_stepK.npy, K=0..15, float32 [256] | promptN_logits_stepK.npy, K=0..9, float32 [151936] |
| logits meaning | logits for generation step K (verified: argmax(stepK) == generated token K) | same |

- N is 1-based (prompt1..prompt3).
- Tiny logits were generated by scripts/build_tiny_logits.py from the existing
  weights and token-verified against reference_outputs.json at every step.
- **Feed the saved input IDs directly. Never tokenize text.** (Dossier §5:
  tokenizer parity is a separate, later problem; a BPE difference here would
  masquerade as a forward-pass bug.)
- No BOS is prepended (dossier §4). The saved IDs are the complete input.
- Generate exactly 256 tokens (4B) / 16 tokens (tiny) with no EOS early-stop —
  that is what the fixtures did.

## Success criteria, in order (do not skip ahead)

1. **Tiny step-0 logits**: prefill prompt1, dump logits, diff vs
   oracle/prompt1_logits_step0.npy. Argmax must match (token 3). Record
   max-abs-diff — bit-exactness is NOT expected (BF16, different accumulation
   order); argmax match at every step IS required.
2. **Tiny full decode**: 3 prompts x 16 tokens, token-exact vs
   reference_outputs.json. All 48 step logits available for bisecting.
3. **4B step logits**: prompt1 steps 0..9 argmax-match.
4. **4B full decode**: 3 prompts x 256 tokens token-exact = **G1a-1 PASS**.
   Record in bench/1a.md.

If step 1 fails: bisect with per-layer activation dumps (write
scripts/dump_tiny_activations.py with torch forward hooks; dump Mojo
activations at the same cut points; first divergent layer is the bug). This
is the "layer-by-layer diffing harness" SPEC.md Phase 1b wants — building it
now is in-gate because it is the G1a-1 debugging tool.

## The forward pass, op by op

Decode-step shapes given for the 4B model; prefill processes seq_len tokens.
All matmuls: `gemm_kernel_rdna` (linalg/matmul/gpu/amd_rdna/matmul.mojo:66),
transpose_b=True, B = weight [out, in] straight from the arena (zero-copy,
DECISIONS 2026-07-11). Accumulator is float32 via get_accum_type.

| # | Op | Kernel / approach | Shapes (decode) |
|---|---|---|---|
| 1 | Embedding lookup | Row copy from embed_tokens (check nn/ for a gather kernel first; for decode it is one 2560-elem row copy — enqueue_copy suffices) | [1, 2560] |
| 2 | Input RMSNorm | rms_norm_gpu (nn/normalization.mojo:1589), eps 1e-6 | [1, 2560] |
| 3 | Q/K/V projections | gemm x3 | Q [1,4096], K [1,1024], V [1,1024] |
| 4 | Per-head QK RMSNorm | rms_norm over head_dim=128 per head, weight [128] | Q as [32,128], K as [8,128] |
| 5 | RoPE | apply_rope (nn/rope.mojo:66), **split-half** indices via get_safetensors_idx (nn/rope.mojo:52) | Q, K per head |
| 6 | Attention | AttentionRDNA (nn/attention/gpu/amd_rdna/attention.mojo:191): mha_prefill / mha_decode; scale = 1/sqrt(128); causal mask | out [1, 4096] |
| 7 | O projection | gemm, weight [2560, 4096] | [1, 2560] |
| 8 | Residual add | elementwise add (trivial; check for vendored elementwise first) | [1, 2560] |
| 9 | FFN RMSNorm | rms_norm_gpu | [1, 2560] |
| 10 | SwiGLU | gate gemm [9728,2560] + up gemm [9728,2560] → silu(gate) * up → down gemm [2560,9728] | [1, 2560] |
| 11 | Residual add | as #8 | [1, 2560] |
| 12 | (after 36 layers) Output RMSNorm | rms_norm_gpu, model.norm.weight | [1, 2560] |
| 13 | LM head | gemm with **embed_tokens as B** [151936, 2560], transpose_b=True | logits [1, 151936] |
| 14 | Greedy sample | argmax — GPU (nn/argmaxmin_gpu.mojo:61) or copy 600KB fp32 to host and argmax there; either is fine at this phase | token id |

### Op order inside attention — verified from HF transformers 5.13.1 source

```
q = q_norm( q_proj(x).view(*, heads, head_dim) )   # norm AFTER projection+reshape
k = k_norm( k_proj(x).view(*, kv_heads, head_dim) )
v =         v_proj(x).view(*, kv_heads, head_dim)  # NO norm on V
q, k = apply_rope(q, k)                             # RoPE AFTER qk-norm
attn -> [*, heads*head_dim] -> o_proj
```

QK-norm BEFORE RoPE. V is never normed. Get this order wrong and every
logit is subtly wrong with no error anywhere.

## The four traps (dossier warnings, at the point they bite)

1. **Non-square projections (op 3, 6, 7).** head_dim is decoupled from
   hidden/heads: Q output is 32x128=4096 ≠ hidden=2560. The attention output
   entering o_proj is [*, 4096], not [*, 2560]. Any code that assumes
   projection output == hidden size produces silent shape garbage. Tiny is
   square-ish by accident (4x32=128=hidden) — **tiny will NOT catch this
   trap; only the 4B logit diff will.** Do not treat tiny-pass as proof here.
2. **QK norm (op 4).** RMSNorm weight is [128] = head_dim, applied per-head
   over the last axis. It is NOT a [4096] norm over the flat Q vector. eps
   1e-6. Order: after projection, before RoPE (verified above).
3. **Split-half RoPE (op 5).** safetensors/HF layout: real parts in the first
   half of head_dim, imaginary in the second half (rotate_half), NOT
   interleaved pairs. MAX's apply_rope supports both — pass interleaved=False
   semantics via get_safetensors_idx. theta differs per model: **4B = 5e6,
   tiny = 1e4** (both from config.json; must be a parameter, and a wrong
   theta still argmax-matches early steps sometimes — diff logits, not vibes).
4. **Tied embeddings (op 13).** There is no lm_head tensor anywhere (loader
   guarantees it). The LM head matmul uses the embed_tokens TileTensor
   [151936, 2560] directly as the transpose_b B operand — zero-copy, no
   transposition, no second buffer.

## KV cache (this phase only)

Single stream, contiguous, pre-allocated for max 256+12 positions. K and V
per layer: [kv_heads, max_seq, head_dim] BF16 in one DeviceBuffer. The
attention kernels take K/V via the MHAOperand trait — a minimal wrapper over
the contiguous buffer must satisfy it (kernel-interface-report §10 says a
trivial wrapper should work; this is Experiment 2 below if it doesn't).
No paging, no batching — that is Phase 2 (SPEC.md).

## Open questions — resolve by experiment FIRST, ~15 min each, log to DECISIONS.md

1. **Kernel launch path.** Is calling the high-level `matmul()` dispatcher
   viable from our code, or do we launch `gemm_kernel_rdna` directly via
   ctx.enqueue_function? Minimal experiment: one [4,2560]x[2560,4096] gemm on
   the R9700 against a torch reference. This also settles the
   TileTensor-for-activations question (dynamic seq_len dim: row_major with
   runtime first dim per environment-notes).
2. **MHAOperand for a contiguous cache.** Read the trait, write the minimal
   wrapper, compile against mha_decode. If a trivial wrapper does not satisfy
   `block_paged_tile`, decide: single-block paged layout vs vendoring more of
   kv_cache.mojo. Do not silently expand scope past what mha_decode needs.
3. **rms_norm_gpu contract.** Which variant dispatches for [1, 2560] and
   [32, 128] shapes; confirm eps parameter and weight layout.
4. **Embedding/gather kernel.** Does nn/ ship one? If not, decode needs only
   an enqueue_copy row copy; prefill needs a small gather loop — write the
   minimal thing, it is a copy, not math.

## Definition of Done

1. scripts/diff_logits.py committed: compares a Mojo-dumped logits .npy (or
   raw f32 file) against an oracle .npy; prints max-abs-diff + argmax match.
2. Tiny: step-0 logits argmax matches on all 3 prompts; 16-token greedy
   decode token-exact on all 3 prompts; max-abs-diff recorded.
3. 4B: steps 0-9 argmax match; 3 x 256 greedy token-exact. **G1a-1 PASS**
   recorded in bench/1a.md with exact commands.
4. Decode loop reuses the KV cache (no re-prefill per token) — otherwise
   G1a-2 is unreachable later.
5. No FP32 weight copies anywhere; weights stay BF16 in the arena (the BF16
   path is permanent, it is the 1b numerics reference).
6. Each working step committed: gemm experiment → rmsnorm parity → one full
   layer parity → tiny logits match → tiny decode → 4B.

## Explicitly out of scope (SPEC.md non-goals, phase 1a)

Tokenizer (fed IDs only), sampling beyond greedy argmax, EOS handling,
HTTP/serving, batching, prefill perf tuning, FP8 anything, second archs.
Do not optimize decode past 90% of llama.cpp F16 (55.14 tok/s baseline →
gate is 49.63): correctness first; G1a-2 tuning is a separate pass.
