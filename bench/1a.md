## G1a-1 — correctness (PASS, three-part gate — see `SPEC.md` §3 for why)

### Methodology

Two checks, both against the committed oracle (`fixtures/oracle/`,
HF transformers 5.13.1, `attn_implementation="sdpa"`, greedy, no BOS,
saved input token IDs fed directly — no re-tokenization):

1. **Full autoregressive decode.** 3 prompts, our own KV cache, our own
   sampled tokens fed back at every step (standard generation) — 256 tokens
   each. Compared token-for-token against the oracle's saved output IDs.
2. **768-step teacher-forced check.** For each prompt: input =
   `prompt_tokens + oracle_output[:255]` (the reference's own generated
   tokens, not ours), run as a **single forward call** over the whole
   sequence. Causal attention means row `(prompt_len-1+k)` of that one
   forward pass gives exactly the logits that predict `oracle_output[k]`, for
   all `k` in `0..255` simultaneously — equivalent to 256 sequential
   teacher-forced decode steps at roughly 1/100th the wall-clock cost. This
   isolates single-step correctness from autoregressive error propagation:
   every one of the 768 steps starts from a history that is correct by
   construction.

   HF's own logits for the identical construction (single forward call, same
   input, same `sdpa`) were computed the same way for an apples-to-apples
   diff — this was necessary because, as below, HF's single-shot batched
   recompute does not reproduce its own cached-autoregressive oracle exactly
   either.

Commands:
```
pixi run mojo build -I ~/projects/modular/max/kernels/src -I src src/qwen_generate.mojo -o /tmp/qwen_gen
pixi run mojo build -I ~/projects/modular/max/kernels/src -I src src/qwen_teacher_forced_full.mojo -o /tmp/qwen_tf_full

python3 scripts/dump_prompt_ids.py <1|2|3> /tmp/pN_ids.txt
/tmp/qwen_gen /tmp/pN_ids.txt /tmp/outN.txt 256
python3 scripts/check_g1a1.py <N> /tmp/outN.txt

/tmp/qwen_tf_full /tmp/pN_ids.txt /tmp/oracle_fullN.txt /tmp/tfN
python3 scripts/hf_teacher_forced_full.py <N> /tmp/hftfN
```

### Result 2: HF is not self-consistent (methodological finding, not ours)

Computing HF's logits the identical way we compute ours (single forward call
over the full teacher-forced sequence) and checking HF's own argmax against
its own cached-autoregressive oracle output:

| prompt | HF self-agreement (batched vs its own cached generation) |
|---|---|
| 1 | 255/256 |
| 2 | 250/256 |
| 3 | 252/256 |

Batched (no KV-cache reuse across the whole sequence in one call) vs.
incrementally-cached attention rounds differently *within HF itself*. This is
why "token-identical vs HF" is retired as the gate (`SPEC.md` §3): there isn't
one HF output to be identical to.

### Result 3: 768-step teacher-forced check — the decisive one

Apples-to-apples: Hephaestus vs. HF computed the same way (single-shot,
`sdpa`, same input construction).

| prompt | mismatches | full-vocab max_abs_diff |
|---|---|---|
| 1 | 1/256 | 12.0621 (step 67, vocab id 96874) |
| 2 | 5/256 | 5.1435 (step 218, vocab id 21132) |
| 3 | 7/256 | 1.7485 (step 205, vocab id 33464) |
| **total** | **13/768** | — |

Every one of the 13 mismatches, classified by HF's own top-1/top-2 gap at
that step:

| prompt | exact ties (gap = 0.0) | 1-bf16-ulp near-ties | clear non-tie flips |
|---|---|---|---|
| 1 | 1 | 0 | **0** |
| 2 | 3 | 2 | **0** |
| 3 | 7 | 0 | **0** |

**Zero non-tie argmax flips across all 768 steps.** The 1-ulp gaps measured
(0.0625 at magnitude ~12, 0.125 at magnitude ~21–25) match
`2^(floor(log2|top1|)) × 2^-7` exactly — the bf16 ulp at that magnitude, not
an approximation.

### The max_abs_diff numbers, with the investigation behind them

12.06 reported bare would be alarming. Before including it: it occurs on
vocab id 96874, which HF itself ranks **2281st** at that step (i.e., a token
HF considers wildly improbable), and **only at that one step** — the same
token elsewhere in the same prompt differs from HF by ~0.075, ordinary noise.
Per-step max_abs_diff sampled every 8 steps across all three prompts shows a
gentle background (mean 0.02–0.1, growing slowly) consistent with ordinary
bf16 accumulation over 36 layers, plus five isolated spikes that don't fit
that trend (12.06, 5.14, 4.15, 1.75, 1.55).

One candidate root cause was tested and ruled out directly: GPU `cos`/`sin`
precision at the large raw angles RoPE's highest-frequency term produces late
in a sequence (the θ⁰=1 rotation pair's angle equals token position
directly, up to ~265 radians here). Measured against Python's exact
`math.cos`/`sin` (`experiments/exp6_gpu_trig_probe.mojo`): GPU error is
1e-6 to 1e-8 — six orders of magnitude too small to explain a 12-unit logit
gap.

**Root cause of the spikes is not identified.** This does not block G1a-1
(see `SPEC.md` §3(c): the bound that matters is at the decision boundary, and
these spikes are argmax-irrelevant tail-token events). It **is** a
precondition for Phase 1b — logged as a Phase 1b entry gate in `DECISIONS.md`
2026-07-12, since FP8's 3-bit mantissa gives any latent numerical fragility
far less room to hide.

**G1a-1 verdict: PASS**, against the three-part gate in `SPEC.md` §3.

## G1a-2 — decode throughput

Target: ≥ 49.63 tok/s (90% of the llama.cpp F16 GGUF baseline, 55.14 tok/s,
`bench/0-baseline-llamacpp-f16.md`). Prompt 1 (10-token prompt, G1a-1 PASS),
3 reps, GPU 0 idle before each run (checked via `rocm-smi`).

Command (exact):
```
pixi run mojo build -I ~/projects/modular/max/kernels/src -I src src/qwen_generate.mojo -o /tmp/qwen_gen
/tmp/qwen_gen /tmp/p1_ids.txt /tmp/out.txt 256
```

### Baseline — naive matmul on every linear projection, including decode

| rep | tok/s |
|---|---|
| 1 | 11.6335 |
| 2 | 11.5919 |
| 3 | 11.5836 |
| **avg** | **11.603** |

Effective bandwidth = 11.603 tok/s × 8.04 GB/tok = 93.3 GB/s = **18.2%** of
the R9700's 512 GB/s peak (spec, `DECISIONS.md` 2026-07-11). Nowhere near
bandwidth-bound: `matmul_kernel_naive`'s 16×16 thread block was built for
tiled 2D problems, and at decode's M=1 shape, 15 of every 16 threads compute
nothing — most of the block is idle.

### Optimization: route decode (M=1) through the vendored `gemv_gpu`

One change: `hephaestus.kernels.linear()` now dispatches to `gemv_gpu`
(`linalg.gemv`, transpose_b=True) when `m == 1`, instead of
`matmul_kernel_naive`. No WMMA involved — `gemv_gpu` is one of the two
kernel paths already proven correct on gfx1201 (`exp3e`,
`DECISIONS.md` 2026-07-12); this is "vendor, don't write," applied to the
decode shape it was actually designed for. Prefill (M>1) is untouched.

Re-verified G1a-1 still holds after the change: prompt 1, 256/256
token-exact.

| rep | tok/s |
|---|---|
| 1 | 47.9996 |
| 2 | 47.9449 |
| 3 | 47.9854 |
| **avg** | **47.977** |

Effective bandwidth = 47.977 × 8.04 GB = **386.0 GB/s = 75.4%** of peak —
up from 18.2%. **4.13× speedup from one substitution.** 96.7% of the 49.63
tok/s target.

### Three further kernel-launch-count reductions (all vendor-none-added,
### each re-verified against the correctness gate before being kept)

The remaining gap was pursued as fixed per-kernel-launch overhead, not
bandwidth headroom (75.4% utilization already, not near a wall). Original
per-layer launch count: 19. Three changes, in order, each mechanical
(no closures except one, no cross-layer assumptions):

1. **K/V projections write directly into the KV cache**, instead of into a
   scratch buffer that a separate `cache_write` kernel then copies. Removes
   2 launches/layer (72 total). `-> 48.46 tok/s` (3 reps: 48.457, 48.453,
   48.457).
2. **Residual add fused into `o_proj`/`down_proj` via the matmul kernels'
   `elementwise_lambda_fn` epilogue** (a read-modify-write closure invoked
   once per output element — every call site in `gemv.mojo` and
   `matmul_kernel_naive` was read first to confirm it always invokes at
   `width=1`, scalar, before relying on that). Removes 2 launches/layer (72
   total). `-> 48.85 tok/s` (3 reps: 48.866, 48.844, 48.848).
3. **Q and K RoPE fused into one kernel launch** (a single kernel branches
   on which half of the combined index range a thread falls in and picks
   the Q or K pointer/head-count accordingly — mechanical, no new closures).
   Removes 1 launch/layer (36 total). `-> 49.13 tok/s` (3 reps: 49.139,
   49.144, 49.104).

Total: 5 fewer launches/layer (19 → 14), **144 + 36 = 180 launches removed
per decode step**.

**Correctness re-verified after every change**, not assumed — the fusions
touch the decode (`M=1`, `gemv_gpu`) path specifically, which the original
768-step teacher-forced check (a single big prefill, `M>1`) does not
exercise at all, so a new **decode-path** teacher-forced harness
(`src/qwen_teacher_forced_decode.mojo`) was written: it steps through 256
real decode calls per prompt, feeding the oracle's own tokens back at each
step (not our argmax), exercising exactly the code path being changed.

- Tiny: token-exact on all 3 prompts × 16 tokens after every change.
- Prompt 1 (4B): 256/256 token-exact, full autoregressive, after every
  change.
- Decode-path teacher-forced (all 3 prompts × 256 steps, post-fusion):
  256/256, 253/256, 250/256 — 9/768 mismatches. **Every one checked
  individually** against HF's own logits at that exact teacher-forced step:
  5 exact ties (gap = 0.0), 4 at exactly one bf16 ulp (gap = 0.125, matching
  `2^(exponent-7)` precisely). **Zero non-tie flips.** The fusions shifted
  *which* existing near-ties get hit (expected — any reordering of
  floating-point operations can do that) but introduced no new class of
  divergence.

**One optimization attempted and rejected before being used.** `gate_proj`
and `up_proj` are byte-contiguous in the staged arena for layer 0
(`gate_end == up_offset` exactly), which would let a single matmul compute
both at once. **Checked across all 36 layers before relying on it — layer
35 is not contiguous** (shard-boundary artifact of staging). Using this
fusion without that check would have silently corrupted layer 35's
`up_proj` output. Not implemented.

### A fourth attempt: fused QK-norm — built, verified correct in isolation,
### reverted anyway

A custom kernel fusing Q-norm and K-norm into one launch (same index-range
branch pattern as the RoPE fusion, since `rms_norm_gpu` can't be used here —
it takes one gamma tensor per call, and Q/K use different weight vectors)
was built and measured at **~49.2–49.3 tok/s**. Every individual mismatch
this introduced was re-verified the same way as every prior change: all 10
disagreements found in the decode-path teacher-forced check were exact ties
or exactly one bf16 ulp, zero non-tie flips.

But it had a side effect the per-mismatch check doesn't see: it changed
*which* prompt hits its near-ties, and **prompt 1 — the prompt supplying
the "one full clean 256-token run" evidence for G1a-1 part (b) — started
diverging at step 4** under the new arithmetic order. With this fusion, none
of the 3 prompts achieved a full clean autoregressive run. Individually
correct, collectively it broke a specific, already-PASSing piece of G1a-1
evidence. **Reverted** rather than re-litigate G1a-1's evidence for a ~0.15
tok/s gain. Logged to `IDEAS.md` as a Phase 2 candidate instead — batching
changes the reduction order anyway, and per-request reduction-order
sensitivity to a single prompt's tie-breaking stops being the load-bearing
concern it is for a single-stream gate.

### Verdict

**89.1% of llama.cpp's raw 55.14 tok/s; 99.0% of the 90%-of-llama.cpp
target (49.63) — 49.13 tok/s, short by 0.50 tok/s (1.0%).** G1a-2 stays
**unchecked**. The gap is characterized, not mysterious: three verified
launch-count reductions (19 → 14 launches/layer) closed most of it, and the
remainder is very likely fixed per-kernel-launch overhead rather than
bandwidth headroom (75%+ utilization already). The path to closing it is
real profiling to find the next-largest per-step cost — a different kind of
work than the launch-count-counting done this session, and not attempted
here.
